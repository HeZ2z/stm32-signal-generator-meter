# LCD Integration Notes

## 当前基线

当前 `ALIENTEK 4342 RGBLCD` 集成以 Apollo `STM32F429IGT6` 板级资料为准，不再沿用其他开发板或同系列面板的经验参数。

当前代码基线：

- `MCU`: `STM32F429IGT6`
- `LCD`: `ALIENTEK 4342 RGBLCD`
- `Framebuffer`: `0xC0000000`
- `SDRAM`: `FMC Bank1`
- `Pixel format`: `RGB565`
- `UART`: 保留为主调试通道

## Apollo 硬件真值

来自 `阿波罗V2 硬件参考手册_V1.0（F429版本）.pdf`：

- `PB5 = LCD_BL`
- `PB6` 未连接板载设备，可继续作为 `PWM_OUT`
- `PC0 = FMC_SDNWE`
- `PC2 = FMC_SDNE0`
- `PC3 = FMC_SDCKE0`
- `PF11 = FMC_SDNRAS`
- `PG15 = FMC_SDNCAS`
- `PG8 = FMC_SDCLK`
- `PG4 = FMC_BA0`
- `PG5 = FMC_BA1`

直接结论：

- `SDRAM` 必须走 `Bank1`
- 帧缓冲基地址必须使用 `0xC0000000`
- `PB5` 不能再作为测量输入，回环固定改为 `PB6 -> PA7`

## Apollo LTDC 参数

来自 `STM32F429 阿波罗开发指南V1.1.pdf` 第 444 到 475 页，对 `0x4342` 面板的参数如下：

- 分辨率：`480x272`
- 像素时钟：`9MHz`
- `HSW = 1`
- `VSW = 1`
- `HBP = 40`
- `VBP = 8`
- `HFP = 5`
- `VFP = 8`
- `HSPolarity = AL`
- `VSPolarity = AL`
- `DEPolarity = AL`
- `PCPolarity = IPC`
- 显存地址：`0xC0000000`

当前实现按这组参数初始化 LTDC，不再使用之前试错阶段的 `41/2/2/10/2/2` 屏参。

## 触摸联动基线

同一套 Apollo HAL 触摸例程表明，`0x4342` 对应的 4.3 寸模组应按 `GT9xxx` 电容触摸路径接入：

- 控制器：`GT9xxx`
- 接口：GPIO 模拟 `I2C`
- 接线：
  - `PH6 = SCL`
  - `PI3 = SDA`
  - `PI8 = RST`
  - `PH7 = INT`

当前代码使用 GPIO 模拟 `I2C` 轮询方式采样，不引入额外硬件 `I2C` 复用风险。

## 当前 LCD 页面

当前 LCD 后端已收敛为更稳定的两阶段页面：

1. 开机四色测试图
2. `M8` 单通道真实采样波形页

当前波形页提供：

- 顶部 `INPUT / OUTPUT` 卡
- 中央真实采样 `YT` 波形框
- 底部频率/占空比调参按钮
- `RESET` 与 `MORE` 按钮
- 最近一次触摸坐标和底部状态提示

当前 `UART` 保留为可写兜底入口，可继续使用 `help`、`status`、`freq <hz>`、`duty <1-99>`。

## M8 波形页收敛结论

当前 `M8` 页面已经固定以下语义：

1. 左侧 `INPUT` 卡固定表示 `PA0 ADC1` 真实采样输入。
2. 右侧 `OUTPUT` 卡固定表示当前 `PB6 TIM4` 输出参数。
3. 中央波形图固定由 `ADC1 + DMA` 真实采样驱动，不使用软件预览波形。

当前显示策略已经收敛为：

- 频率窗内优先显示真实输入 `F/D`
- 超出占空比稳定估算窗口时显示 `F=<当前输出频率> D=--`
- 断线或无效输入时退化为无效输入状态
- `ADC LIVE` 仅保留给短时边沿不完整窗口，不再作为常态显示

当前实板为降低误判，已同时采用：

- `PA0` 下拉，避免拔线后悬空漂移
- 更严格的方波存在判定，避免随机噪声被解释为有效输入
- 输入估算短保持和失败滞回，减少偶发 `ADC LIVE`

当前已确认的观感结果：

- 波形刷新频闪已压制到可接受范围
- 顶部输入/输出卡与当前页面语义一致
- 拔掉 `PB6 -> PA0` 后不再显示随机 `F/D`
- 偶发 `ADC LIVE` 已显著减少，但当前实现仍优先保证单缓冲 CPU 绘制路径简单可解释

## 触摸问题收敛结论

本轮联调后，触摸问题已经收敛出 3 个关键事实：

1. `4342` 的触摸必须走 `GT9xxx/GT9147` 电容触摸路径，不能再按电阻触摸实现。
2. GPIO 模拟 `I2C` 需要放慢到接近 Apollo HAL 例程的时序量级，否则控制器虽然可能被识别，但后续读写不稳定。
3. `GT9xxx` 的状态位必须在 `status & 0x80` 时按例程及时清除；此前“触摸 ready 但始终无反应”的根因就是空闲 `0x80` 状态未被清除，阻塞了后续真实触点上报。

当前实板已能看到：

- `GT9XXX READY ID=9147`
- 触摸后串口 `SET` 参数发生变化
- LCD 触摸按键可驱动参数调整

## 当前实现策略

当前 `display_lcd` 实现采用最小联调路径：

1. 初始化 `PB5` 背光 GPIO，默认先关背光
2. 初始化 `SDRAM`
3. 对 `0xC0000000` 做最小读写探针
4. 初始化 `LTDC` 和单层 `RGB565` layer
5. 写入四色测试图案
6. 点亮背光
7. 后续进入 `M8` 单通道真实采样波形页
8. 触摸按键直接修改当前 `PWM` 频率和占空比

失败时不让 LCD 故障拖垮主功能，串口仍持续输出：

- `sdram init failed`
- `sdram rw failed`
- `ltdc init failed`

## 已知失败模式

到本轮修正前，已经确认过两类错误来源：

1. `Bank2 + 0xD0000000`
- 会直接导致 `LCD sdram rw failed`
- 与 Apollo 板级接线不符

2. 错误的 SDRAM MSP 引脚假设
- 把 Discovery 风格的 `PB5/PB6` FMC 管脚带进来后，会干扰 Apollo 实板行为
- 还可能连带影响主业务引脚和测量稳定性

本轮修正后，如果 `SDRAM` 探针通过但仍未正常显示，后续问题范围应收缩为：

- LTDC 同步极性
- LTDC 时钟
- 背光时序
- RGB 数据线或面板连接状态

不再优先怀疑主业务的 `UART/PWM/MEAS` 链路。

## 实板验证步骤

1. 构建并烧录当前固件
2. 接回环线：`PB6(TIM4_CH1) -> PA7(TIM3_CH2)`
3. 打开串口：
   - `python3 tools/serial_monitor.py --port /dev/ttyUSB0 --baud 115200`
4. 先观察上电后的 `LCD` 状态：
   - 目标：不再出现 `LCD sdram rw failed`
5. 再验证 3 组推荐演示参数：
   - 默认 `1000/50`
   - `freq 2000` + `duty 30`
   - `freq 5000` + `duty 70`
6. 使用 LCD 触摸按键调整参数，确认串口 `SET` 跟随变化
7. 拔掉回环线，确认 `MEAS no-signal`

## 推荐演示边界

- 推荐主演示：`1000/50`、`2000/30`、`5000/70`
- `20Hz` 可用于展示低频切换时状态更新节奏
- `100000Hz` 可用于展示 `1 MHz` 计时基准下的量化误差，不建议作为主演示

## 当前结论

当前项目的 M6 主线已经从“猜测性 LCD 接入”收敛到“Apollo F429 手册真值驱动的 LCD 联调”。后续实板排障只需要围绕这套固定基线继续推进，不应再切回 `Bank2 + 0xD0000000` 或其他未证实的板级假设。
